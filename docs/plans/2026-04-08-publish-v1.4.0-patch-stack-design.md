# Publish v1.4.0 Patch Stack Design

**Date:** 2026-04-08
**Status:** Approved

## Goal

Publish the refreshed `v1.4.0` patch stack with the fastest safe manual release path, then update `workstation` to consume the new published `opencode-patched` release.

## Ownership

- `~/projects/opencode-cached`
  Owns `patches/caching.patch` and publishes `v1.4.0-cached`.
- `~/projects/opencode-patched`
  Owns `patches/vim.patch` and `patches/tool-fix.patch`, consumes `opencode-cached`, and publishes `v1.4.0-patched`.
- `~/projects/workstation`
  Consumes published `opencode-patched` releases by version and artifact hash.

`opencode-vim` is not part of the active release chain for this publish.

## Publish Path

The fastest safe path is a direct manual chain:

1. Confirm `opencode-cached` and `opencode-patched` `main` branches contain the refreshed patch commits.
2. Manually dispatch `opencode-cached` `build-release.yml` for `version=1.4.0`.
3. Watch the run to completion and confirm release `v1.4.0-cached` exists.
4. Manually dispatch `opencode-patched` `build-release.yml` for `version=1.4.0`.
5. Watch the run to completion and confirm release `v1.4.0-patched` exists.
6. Update `workstation` to use `1.4.0` and the new artifact hashes.

## Why This Path

- It avoids waiting for scheduled workflows.
- It avoids an extra hop through `sync-cached.yml` when the target version is already known.
- It makes failures obvious at the exact stage they occur.

## Verification Gates

Before triggering releases:

- `opencode-cached` and `opencode-patched` `main` must be pushed.
- Local patch-stack verification against fresh upstream `v1.4.0` must already be green.

After cached publish:

- GitHub release `v1.4.0-cached` must exist.
- Release artifacts must be present.

After patched publish:

- GitHub release `v1.4.0-patched` must exist.
- Release artifacts must be present.

Before updating workstation:

- Determine new hashes for all four release artifacts from the published `opencode-patched` release.
- Update the pinned `version` and `hash` values in `users/dev/home.base.nix`.

## Risks

### CI-only Failure

Local verification already passed, but GitHub Actions rebuilds from scratch on Linux and macOS. A workflow can still fail because of runner differences, network issues, or release packaging differences.

### Cached Release Visibility

`opencode-patched` depends on the cached release existing before its build is triggered. The patched build should not start until `v1.4.0-cached` is visible on GitHub.

### Workstation Hash Drift

`workstation` cannot be updated until the final `opencode-patched` release exists because the artifact hashes are release-specific.

## Non-Goals

- Do not route publishing through `opencode-vim`.
- Do not wait for scheduled automation when manual dispatch is faster.
- Do not update `workstation` before `v1.4.0-patched` exists.
