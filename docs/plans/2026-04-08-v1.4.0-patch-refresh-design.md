# v1.4.0 Patch Refresh Design

**Date:** 2026-04-08
**Status:** Approved
**Upstream target:** `anomalyco/opencode` `v1.4.0`

## Goal

Refresh the patch stack for OpenCode `v1.4.0` by rebasing each patch onto current upstream behavior instead of mechanically preserving older diffs.

## Repositories and Ownership

- `~/projects/opencode-patched`
  Owns `patches/vim.patch`, `patches/tool-fix.patch`, stack verification, and release wiring.
- `~/projects/opencode-cached`
  Owns `patches/caching.patch`.
- Other `~/projects/opencode-*` repos
  Own any patches they publish. This repo should consume those outputs rather than duplicate ownership.

`opencode-patched` remains the center of patch-stack work and documentation. Repo-local guidance in `.opencode/skills` should explain the ownership model so future refreshes happen in the correct repository.

## Source of Truth by Patch

### Tool Fix

- First verify whether upstream `v1.4.0` already contains the `tool_use/tool_result` mismatch fix from PR `#16751`.
- If upstream already contains the behavior, drop `patches/tool-fix.patch`.
- If upstream does not, refresh `patches/tool-fix.patch` from the current PR behavior and keep only the minimal patch needed.

### Vim

- Use PR `#12679` as the behavioral guide, not the old local patch as the canonical source.
- Rebase vim behavior onto the current `v1.4.0` TUI/input/config structure.
- Include any required schema or docs changes that are now part of carrying the feature on top of `v1.4.0`.

### Caching

- Continue owning and updating `patches/caching.patch` in `~/projects/opencode-cached`.
- Rebase the caching behavior onto `v1.4.0` provider/prompt code rather than preserving old hunk structure.

## Recommended Refresh Order

1. `tool-fix.patch`
2. `vim.patch`
3. `caching.patch`
4. Full-stack verification in a fresh upstream `v1.4.0` checkout

This order gives a quick read on whether one patch can be dropped, then refreshes the lighter local patch before the larger caching rebase.

## Validation Strategy

For each patch:

1. Start from a fresh upstream `v1.4.0` checkout.
2. Write or repair the targeted regression test first.
3. Verify the test fails on plain upstream when the behavior is missing.
4. Implement the minimal port needed to restore the behavior.
5. Verify targeted tests pass.
6. Regenerate the patch from the resulting clean diff.

After all patches are refreshed:

1. Apply `caching.patch` from `opencode-cached`.
2. Apply local `vim.patch`.
3. Apply local `tool-fix.patch` if it still exists.
4. Run `typecheck` and patch-specific targeted tests.

## Documentation Changes

Documentation updates should happen in `opencode-patched`:

- Update `README.md` to make patch ownership and refresh responsibilities explicit.
- Add `.opencode/skills` guidance describing:
  - which repo owns which patch,
  - when to use upstream PRs as behavioral guides,
  - when to edit patches in sibling `~/projects/opencode-*` repos instead of this one.

## Current Known State

- Local `tool-fix.patch` applies to upstream `v1.4.0`, but its current local test variant is stale relative to the newer upstream PR form.
- PR `#16751` still applies cleanly to `v1.4.0`, so the fix is still portable.
- Local `vim.patch` does not apply cleanly to `v1.4.0` and needs a true rebase.
- `caching.patch` in `opencode-cached` does not apply cleanly to `v1.4.0` and needs a true rebase there.

## Non-Goals

- Do not move caching patch ownership into `opencode-patched`.
- Do not preserve old patch text when upstream structure has clearly moved.
- Do not treat patch-drift alerts as proof of build breakage without fresh verification.
